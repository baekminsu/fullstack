# 객체지향 설계

객체지향은 클래스를 많이 만드는 기술이 아니라 책임을 잘 나누고 협력 관계를 명확히 하는 사고방식입니다.

## 먼저 잡아야 할 개념

- 객체: 상태와 행동을 함께 가진 협력 단위
- 책임: 객체가 알아야 하거나 해야 하는 일
- 캡슐화: 변경 가능성이 높은 결정을 내부에 숨기는 것
- 다형성: 호출자가 구체 타입보다 역할에 의존하게 하는 것
- 의존성: 한 코드가 다른 코드의 변경에 영향을 받는 관계

## SOLID를 공부하는 방식

SOLID를 암기 목록으로 보지 않습니다. 아래 질문으로 코드를 검토합니다.

- SRP(Single Responsibility Principle): 이 모듈은 누구의 요구 때문에 변경되는가?
- OCP(Open Closed Principle): 새 정책을 추가할 때 기존 코드를 얼마나 고치는가?
- LSP(Liskov Substitution Principle): 하위 타입을 넣어도 호출자의 기대가 깨지지 않는가?
- ISP(Interface Segregation Principle): 사용하지 않는 메서드에 의존하고 있지 않은가?
- DIP(Dependency Inversion Principle): 비즈니스 규칙이 프레임워크나 DB(Database) 세부사항에 끌려다니지 않는가?

## 연습 과제

1. 할인 정책, 쿠폰, 포인트를 가진 주문 도메인을 만듭니다.
2. 정책을 조건문으로 구현한 버전과 인터페이스로 분리한 버전을 비교합니다.
3. 새 할인 정책을 추가할 때 변경되는 파일 수를 기록합니다.
4. 테스트가 쉬워진 지점과 오히려 복잡해진 지점을 회고합니다.

## 흔한 실수

- getter와 setter만 있는 데이터 클래스를 객체지향으로 착각합니다.
- 모든 것을 인터페이스로 만들고 실제 변경 가능성은 확인하지 않습니다.
- 도메인 규칙을 서비스 하나에 몰아넣고 엔티티를 빈 구조체로 둡니다.
- 상속으로 재사용하려다가 결합도를 높입니다.

## 체크리스트

- [ ] 객체가 스스로 지켜야 할 불변식을 가지고 있다.
- [ ] 외부에서 객체 상태를 마음대로 깨뜨릴 수 없다.
- [ ] 정책 변경이 조건문 폭발로 이어지지 않는다.
- [ ] 인터페이스는 실제 경계가 필요할 때 만든다.
- [ ] 테스트가 객체의 책임을 설명한다.
