# 추천 도서와 읽는 순서

책은 한 번에 많이 읽기보다 프로젝트와 연결해서 읽어야 오래 남습니다.

## 1순위: 코드와 설계

- 클린 코드
- 리팩터링
- 오브젝트
- 객체지향의 사실과 오해
- 테스트 주도 개발
- Working Effectively with Legacy Code

읽는 방법:

- 한 장을 읽으면 현재 프로젝트 코드에서 적용할 지점을 하나 찾습니다.
- 책의 예제를 그대로 믿지 말고 현재 언어와 프레임워크에 맞게 바꿔봅니다.
- 리팩터링 전후 테스트 결과를 남깁니다.

## 2순위: 아키텍처와 도메인

- 클린 아키텍처
- Domain-Driven Design Quickly
- Implementing Domain-Driven Design
- Patterns of Enterprise Application Architecture
- Software Architecture: The Hard Parts

읽는 방법:

- 모든 패턴을 적용하려 하지 않습니다.
- 현재 프로젝트에서 변경이 잦은 부분과 안정적인 부분을 구분합니다.
- 아키텍처 선택의 장점뿐 아니라 비용을 같이 기록합니다.

## 3순위: CS(Computer Science)와 시스템

- Computer Systems: A Programmer's Perspective
- Operating Systems: Three Easy Pieces
- Computer Networking: A Top-Down Approach
- Designing Data-Intensive Applications
- Database Internals

읽는 방법:

- 개념을 Spring, React, PostgreSQL의 실제 문제와 연결합니다.
- 성능 문제를 직접 재현하고 측정합니다.
- 네트워크, OS(Operating System), DB(Database)가 한 요청 처리에 어떻게 관여하는지 추적합니다.

## 4순위: 프론트엔드와 사용자 경험

- JavaScript: The Good Parts
- Effective TypeScript
- Learning React
- Designing Interfaces
- Refactoring UI

읽는 방법:

- 컴포넌트 설계와 사용자 흐름을 함께 봅니다.
- 접근성, 로딩, 에러, 빈 상태를 디자인의 일부로 다룹니다.
- 성능 최적화는 측정 후 적용합니다.

## 5순위: 운영과 커리어

- Site Reliability Engineering
- The Pragmatic Programmer
- The Mythical Man-Month
- Peopleware
- Staff Engineer

읽는 방법:

- 개인 기술뿐 아니라 팀과 시스템의 반복 문제를 관찰합니다.
- 좋은 의사결정 문서와 나쁜 의사결정 문서를 비교합니다.
- 코드 작성 외에 리뷰, 운영, 멘토링의 기준을 세웁니다.
