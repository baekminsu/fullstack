# 클린 코드 학습 가이드

클린 코드는 예쁜 코드가 아니라 변경 비용이 낮은 코드입니다. 읽기 쉽고, 테스트하기 쉽고, 잘못된 사용을 막는 코드를 목표로 합니다.

## 핵심 질문

- 이 함수는 한 가지 이유로만 변경되는가?
- 이름만 보고 의도를 알 수 있는가?
- 조건문과 예외 흐름이 도메인 규칙을 가리고 있지 않은가?
- 테스트 없이 수정하기 두려운 부분은 어디인가?
- 중복 제거가 오히려 개념을 흐리게 만들지는 않았는가?

## 학습 순서

1. 좋은 이름 짓기
2. 함수 크기 줄이기
3. 조건문 단순화
4. 예외 처리와 실패 모델 정리
5. 테스트 가능한 코드 만들기
6. 리팩터링을 작은 단위로 반복하기

## Spring 예제 과제

- 컨트롤러에 있는 검증, 권한, 변환, 비즈니스 로직을 분리합니다.
- 서비스 메서드가 너무 많은 일을 한다면 유스케이스 단위로 나눕니다.
- 엔티티, DTO(Data Transfer Object), 요청 객체의 책임을 명확히 분리합니다.
- 예외를 무작정 `RuntimeException`으로 던지지 않고 실패 의미를 이름으로 드러냅니다.

## React 예제 과제

- 한 컴포넌트가 데이터 요청, 상태 관리, 렌더링, 폼 검증을 모두 하지 않게 나눕니다.
- 불리언 props가 많아지면 상태 모델을 다시 설계합니다.
- 비동기 로딩, 빈 상태, 에러 상태를 명시적으로 처리합니다.
- 커스텀 훅은 재사용보다 관심사 분리를 위해 먼저 사용합니다.

## 체크리스트

- [ ] 함수 이름이 구현 방식보다 의도를 말한다.
- [ ] 하나의 함수에서 추상화 수준이 섞이지 않는다.
- [ ] 주석은 코드로 표현하기 어려운 이유와 맥락만 설명한다.
- [ ] 테스트 이름이 요구사항을 설명한다.
- [ ] 리팩터링 전후로 동작을 검증할 수 있다.
