# 클린 아키텍처

클린 아키텍처의 핵심은 비즈니스 규칙을 프레임워크, 데이터베이스, UI 세부사항으로부터 보호하는 것입니다. 모든 프로젝트에 복잡한 계층을 강제하는 것이 아니라 변경 방향을 통제하는 데 목적이 있습니다.

## 의존성 방향

안쪽으로 갈수록 중요한 정책입니다.

1. Domain: 엔티티, 값 객체, 도메인 규칙
2. Application: 유스케이스, 트랜잭션 경계, 포트
3. Interface Adapter: 컨트롤러, 프레젠터, 리포지토리 구현
4. Infrastructure: DB, 외부 API, 파일 시스템, 프레임워크 설정

의존성은 바깥에서 안쪽으로 향해야 합니다. 도메인이 Spring, JPA, React, PostgreSQL을 몰라야 오래 버팁니다.

## Spring 적용 기준

작은 프로젝트에서는 과한 패키지 분리가 비용이 될 수 있습니다. 대신 아래 경계를 먼저 지킵니다.

- Controller는 HTTP 요청과 응답 변환에 집중합니다.
- UseCase 또는 Service는 애플리케이션 흐름을 담당합니다.
- Domain은 핵심 규칙과 불변식을 담당합니다.
- Repository 인터페이스는 애플리케이션 또는 도메인 쪽에 두고 구현은 인프라에 둡니다.
- JPA Entity와 Domain Model을 분리할지는 복잡도와 변경 가능성으로 결정합니다.

## React 적용 기준

프론트엔드에서도 같은 원칙을 적용할 수 있습니다.

- UI 컴포넌트는 표시와 사용자 입력에 집중합니다.
- 도메인 계산은 순수 함수나 별도 모듈로 분리합니다.
- 서버 API 호출은 화면 곳곳에 흩뿌리지 않습니다.
- 사용자 흐름 단위로 상태와 에러 처리를 설계합니다.

## 연습 프로젝트 구조

```text
src/main/java/com/example/app
  domain/
  application/
  adapter/in/web/
  adapter/out/persistence/
  infrastructure/
```

처음부터 이 구조를 고집하지 않습니다. 기능이 늘어났을 때 어떤 코드가 어떤 이유로 바뀌는지 관찰한 뒤 경계를 강화합니다.

## 체크리스트

- [ ] 도메인 규칙이 컨트롤러나 DB 쿼리에 숨어 있지 않다.
- [ ] 유스케이스가 한 사용자 목표를 표현한다.
- [ ] 외부 시스템 장애를 애플리케이션 실패 모델로 바꿔 처리한다.
- [ ] 테스트에서 DB 없이 핵심 규칙을 검증할 수 있다.
- [ ] 계층 분리가 이름만 있고 의존성은 뒤섞여 있지 않다.
