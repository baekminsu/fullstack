# 20년 풀스택 개발자 성장 로드맵

이 로드맵은 "책을 많이 읽는 사람"보다 "복잡한 문제를 안정적으로 해결하는 사람"을 목표로 합니다. 기간은 절대값이 아니라 집중 순서입니다.

## 0단계: 기준선 만들기

목표 기간: 1개월

- Java, TypeScript, SQL(Structured Query Language) 기본기를 빠르게 점검합니다.
- Git, GitHub, Linux CLI(Command Line Interface), HTTP(HyperText Transfer Protocol), JSON(JavaScript Object Notation), REST(Representational State Transfer)의 실제 사용 능력을 확인합니다.
- 하나의 작은 CRUD(Create, Read, Update, Delete) 서비스를 Spring, React, PostgreSQL로 끝까지 배포해봅니다.
- 모든 학습 기록은 이 저장소에 Markdown으로 남깁니다.

완료 기준:

- 로컬에서 백엔드, 프론트엔드, DB(Database)를 실행할 수 있습니다.
- API(Application Programming Interface) 명세, ERD(Entity Relationship Diagram), 테스트 결과, 회고를 한 번 이상 작성했습니다.
- 모르는 개념을 "정의, 왜 필요한가, 예시, 주의점" 형식으로 설명할 수 있습니다.

## 1단계: 실무 기본기

목표 기간: 6개월

- Clean Code와 객체지향 원칙을 작은 코드에 반복 적용합니다.
- Spring MVC(Model View Controller), Validation, Security, JPA(Java Persistence API), 트랜잭션을 익힙니다.
- React 컴포넌트 설계, 상태 관리, 폼, 라우팅, 비동기 처리를 익힙니다.
- PostgreSQL의 조인, 인덱스, 실행 계획, 트랜잭션 격리를 익힙니다.
- 단위 테스트, 통합 테스트, E2E 테스트의 용도를 구분합니다.

완료 기준:

- 로그인, 권한, 파일 업로드, 검색, 페이지네이션이 있는 서비스를 만들 수 있습니다.
- 테스트 없이 고치는 것이 위험한 코드와 테스트로 보호할 수 있는 코드를 구분합니다.
- SQL 실행 계획을 보고 느린 쿼리의 원인을 설명할 수 있습니다.

## 2단계: 제품을 책임지는 개발자

목표 기간: 1년

- Clean Architecture를 실제 프로젝트에 과하지 않게 적용합니다.
- 도메인 모델, 유스케이스, 어댑터, 인프라 계층을 분리합니다.
- 장애 로그, 메트릭, 추적 정보를 남기고 원인을 좁히는 훈련을 합니다.
- React에서는 사용자 흐름, 접근성, 성능, 에러 상태까지 설계합니다.
- PostgreSQL에서는 동시성, 잠금, 백업, 마이그레이션을 다룹니다.

완료 기준:

- 기능 추가 요청을 받으면 영향 범위, 위험, 테스트 전략을 먼저 말할 수 있습니다.
- API(Application Programming Interface), DB(Database), UI(User Interface) 변경을 하나의 배포 단위로 계획할 수 있습니다.
- 장애 상황을 재현하고 원인, 수정, 재발 방지까지 문서화할 수 있습니다.

## 3단계: 시스템 설계와 운영

목표 기간: 2년에서 5년

- 단일 서버를 넘어 캐시, 메시지 큐, 검색, 파일 저장소를 다룹니다.
- 분산 시스템의 기본 문제인 일관성, 중복 처리, 재시도, 순서를 이해합니다.
- 대규모 프론트엔드 코드베이스의 모듈 경계와 상태 흐름을 설계합니다.
- 데이터 모델 변경, 무중단 배포, 관측 가능성을 운영 기준으로 배웁니다.

완료 기준:

- 트래픽 증가, 데이터 증가, 조직 증가 중 무엇이 문제인지 구분합니다.
- 시스템 설계안을 장단점, 비용, 운영 리스크와 함께 설명합니다.
- 코드 품질 논쟁을 취향이 아니라 유지보수 비용으로 설명합니다.

## 4단계: 시니어 엔지니어

목표 기간: 5년에서 10년

- 팀의 기술 의사결정 기준을 세우고 문서화합니다.
- 코드 리뷰에서 단순 스타일보다 설계, 테스트, 장애 가능성을 봅니다.
- 요구사항을 제품, 도메인, 데이터, 운영 관점으로 나누어 정리합니다.
- 신규 개발자가 빠르게 온보딩할 수 있는 구조와 문서를 만듭니다.

완료 기준:

- 복잡한 문제를 작고 검증 가능한 단계로 분해합니다.
- 팀이 반복해서 겪는 문제를 도구, 프로세스, 아키텍처로 줄입니다.
- "왜 이렇게 만들었는가"를 미래의 팀원이 이해할 수 있게 남깁니다.

## 5단계: 20년차 수준의 기준

목표 기간: 10년에서 20년

- 기술 선택보다 문제 정의, 비용, 조직 맥락, 장기 유지보수성을 먼저 봅니다.
- 레거시 시스템을 존중하면서 점진적으로 개선합니다.
- 성능, 보안, 장애, 데이터 무결성, 사용자 경험을 함께 고려합니다.
- 사람과 시스템이 동시에 성장할 수 있는 개발 문화를 만듭니다.

완료 기준:

- 모르는 기술도 원리, 문서, 작은 실험으로 빠르게 판단합니다.
- 단기 납기와 장기 품질 사이에서 현실적인 선택지를 제시합니다.
- 직접 코딩, 설계, 리뷰, 멘토링, 운영 개선을 모두 수행할 수 있습니다.
