# Definition of Done

Definition of Done은 작업을 끝냈다고 말하기 위한 최소 기준입니다. 완료 기준이 없으면 작업은 Done이 아니라 중단된 상태일 수 있습니다.

## 공통 완료 기준

모든 작업은 아래를 확인합니다.

- [ ] 작업 목적을 만족한다.
- [ ] 변경 범위가 작업 제목과 맞다.
- [ ] 관련 issue 또는 task 번호가 연결되어 있다.
- [ ] 검증 방법이 남아 있다.
- [ ] 문서가 필요한 변경이면 문서를 업데이트했다.
- [ ] 불필요한 임시 파일, 로그, 로컬 설정이 포함되지 않았다.

## Feature 완료 기준

- [ ] 요구사항이 구현되었다.
- [ ] 성공 케이스가 검증되었다.
- [ ] 실패 케이스가 검증되었다.
- [ ] API(Application Programming Interface) 변경이 있으면 문서화했다.
- [ ] DB(Database) 변경이 있으면 migration과 롤백 관점을 확인했다.
- [ ] UI(User Interface) 변경이 있으면 기본 사용자 흐름을 확인했다.
- [ ] 보안이나 권한 영향이 있으면 확인했다.

## Bug 완료 기준

- [ ] 재현 절차가 있다.
- [ ] 원인을 설명할 수 있다.
- [ ] 수정 후 같은 절차로 재현되지 않는다.
- [ ] 가능한 경우 회귀 테스트가 추가되었다.
- [ ] 비슷한 문제가 생길 수 있는 주변 경로를 확인했다.

## Docs 완료 기준

- [ ] 대상 독자가 명확하다.
- [ ] 문서가 현재 코드나 운영 방식과 맞다.
- [ ] 약어는 첫 언급에서 풀네임을 병기했다.
- [ ] 관련 문서와 링크가 연결되어 있다.
- [ ] 오래된 정보가 섞여 있지 않다.

## Refactor 완료 기준

- [ ] 외부 동작이 바뀌지 않았다.
- [ ] 기존 테스트가 통과한다.
- [ ] 필요한 경우 부족한 테스트를 먼저 추가했다.
- [ ] 변경 후 책임과 의존성이 더 명확하다.
- [ ] 단순 파일 이동이나 이름 변경과 구조 변경을 한 커밋에 과하게 섞지 않았다.

## Performance 완료 기준

- [ ] 개선 전 측정값이 있다.
- [ ] 개선 후 측정값이 있다.
- [ ] 측정 조건이 기록되어 있다.
- [ ] 성능 개선 때문에 가독성, 안정성, 데이터 정합성을 해치지 않았다.
- [ ] DB 쿼리 개선이면 `EXPLAIN ANALYZE` 결과를 확인했다.

## Pull Request 완료 기준

- [ ] pull request 제목이 commit convention과 맞다.
- [ ] Summary가 변경 목적을 설명한다.
- [ ] Related에 issue나 task 번호가 있다.
- [ ] Verification에 실행한 검증이 적혀 있다.
- [ ] Risk에 영향 범위나 주의점이 적혀 있다.
- [ ] 리뷰 피드백을 반영했다.

## GitHub Projects Done 기준

GitHub Projects에서 Done으로 옮기기 전에 확인합니다.

- [ ] main branch에 merge되었다.
- [ ] 관련 pull request가 닫혔다.
- [ ] 관련 issue가 닫혔다.
- [ ] Notion에 남겨야 할 결정이나 회고를 남겼다.
- [ ] 후속 작업이 있으면 새 issue로 분리했다.

## Notion 정리 기준

모든 작업을 Notion에 기록할 필요는 없습니다. 아래에 해당하면 Notion에 남깁니다.

- 설계 결정이 있었다.
- 나중에 다시 볼 학습 내용이 있다.
- 장애나 버그 원인이 중요하다.
- 기술 선택의 장단점이 있었다.
- 포트폴리오에 들어갈 수 있는 프로젝트 결과가 있다.

## 체크리스트

- [ ] Done은 "코드를 다 썼다"가 아니라 검증과 정리까지 끝난 상태다.
- [ ] 작업 종류별 완료 기준을 확인했다.
- [ ] 후속 작업은 현재 작업에 억지로 넣지 않고 새로 분리했다.
- [ ] Notion과 GitHub Projects의 상태가 서로 모순되지 않는다.
