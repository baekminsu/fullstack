# 작업 단위 정의

작업 단위는 도구가 바뀌어도 유지되어야 합니다. Notion, GitHub Projects, Azure DevOps, Jira, Linear는 이름과 UI(User Interface)가 다를 수 있지만, 개발자가 다루는 일의 종류는 크게 다르지 않습니다.

## 공통 작업 단위

| 이름 | 의미 | 예시 | 완료 결과 |
| --- | --- | --- | --- |
| Epic | 여러 feature를 묶는 큰 목표 | 학습 플랫폼 구축 | 여러 feature 완료 |
| Feature | 사용자나 시스템에 새 가치를 주는 기능 | 게시글 검색 추가 | 동작하는 기능 |
| Task | feature를 완성하기 위한 작은 실행 단위 | 검색 API 작성 | 독립 작업 완료 |
| Bug | 기대 동작과 실제 동작이 다른 문제 | 로그인 만료 후 500 오류 | 재현과 수정 |
| Docs | 문서 추가 또는 수정 | Spring 로드맵 보강 | 읽을 수 있는 문서 |
| Spike | 불확실성을 줄이는 조사 작업 | React Query 도입 검토 | 결정 근거 |
| Chore | 기능과 직접 관련 없는 관리 작업 | 의존성 업데이트 | 관리 작업 완료 |
| Refactor | 외부 동작을 유지한 구조 개선 | 주문 서비스 책임 분리 | 테스트 유지 |
| Performance | 성능 개선 작업 | 목록 조회 쿼리 튜닝 | 측정 결과 개선 |

## Epic

Epic은 한 번에 끝내기 어려운 큰 목표입니다.

좋은 Epic:

- 여러 feature로 쪼갤 수 있습니다.
- 목표와 성공 기준이 있습니다.
- 일정과 우선순위 논의에 사용됩니다.

예시:

```text
Epic: 개인 지식 관리 서비스 구축

목표:
- Markdown 노트 작성
- 태그 검색
- 공개/비공개 설정
- 버전 기록
```

## Feature

Feature는 사용자나 시스템 관점에서 의미 있는 기능 단위입니다.

예시:

```text
Feature: 게시글 검색

사용자는 제목과 내용으로 게시글을 검색할 수 있다.
검색 결과는 최신순으로 페이지네이션된다.
```

Feature는 보통 여러 task로 나뉩니다.

## Task

Task는 개발자가 실행할 수 있는 작은 작업입니다.

좋은 Task:

- 하루 안에 끝낼 수 있을 정도로 작습니다.
- 완료 기준이 명확합니다.
- commit 또는 pull request와 연결하기 쉽습니다.

예시:

```text
Task: 게시글 검색 API 작성

완료 기준:
- keyword query parameter를 받는다.
- 제목과 내용을 검색한다.
- 페이지네이션을 지원한다.
- 통합 테스트를 추가한다.
```

## Bug

Bug는 재현 조건이 중요합니다.

Bug에는 아래가 있어야 합니다.

- 현재 동작
- 기대 동작
- 재현 절차
- 영향 범위
- 수정 검증 방법

예시:

```text
Bug: 로그인 만료 후 내 정보 조회 시 500 오류

현재 동작:
- 토큰 만료 후 /api/me 호출 시 500 응답

기대 동작:
- 401 Unauthorized 응답
```

## Docs

Docs는 문서 작업입니다. 문서도 개발 결과물입니다.

예시:

- README 보강
- API(Application Programming Interface) 명세 작성
- 학습 로드맵 정리
- 장애 회고 작성
- ADR(Architecture Decision Record) 작성

## Spike

Spike는 구현보다 조사와 판단이 목표입니다.

좋은 Spike 완료 결과:

- 조사한 선택지
- 장단점
- 추천안
- 다음 action

예시:

```text
Spike: React Query 도입 검토

결과:
- 서버 상태 캐싱에는 장점이 크다.
- 현재 프로젝트 규모에서는 우선 직접 fetch wrapper로 시작한다.
- 목록/상세 캐시가 복잡해지는 시점에 도입한다.
```

## 작업 분해 기준

작업이 너무 큰 신호:

- 완료 기준이 5개 이상입니다.
- 백엔드, 프론트엔드, DB(Database), 배포가 모두 섞여 있습니다.
- 하루 안에 끝날지 예측하기 어렵습니다.
- 리뷰어가 한 번에 이해하기 어렵습니다.

작게 나누는 기준:

- API
- UI
- DB migration
- 테스트
- 문서
- 운영 설정

## 체크리스트

- [ ] 작업 이름만 보고 목적을 알 수 있다.
- [ ] 작업 종류가 Feature, Task, Bug, Docs, Spike 중 하나로 분류된다.
- [ ] 완료 기준이 검증 가능한 문장으로 적혀 있다.
- [ ] 관련 issue 번호가 있으면 commit과 pull request에 연결한다.
- [ ] 도구가 바뀌어도 작업 의미가 유지된다.
